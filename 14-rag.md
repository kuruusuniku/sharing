# 14. RAGとベクトルDB

LLMは膨大な知識を持っていますが、**すべてを知っているわけではありません**。社内ドキュメント、最新のリリースノート、自社の製品仕様——こうした情報はLLMの学習データに含まれていません。RAG（Retrieval-Augmented Generation）は、この問題をシンプルに解決するアプローチです。

## なぜRAGが必要なのか

### LLMの知識の限界

LLMには根本的な制約があります。

- **学習データのカットオフ** — モデルの学習が完了した時点以降の情報は知らない
- **社内情報を知らない** — 社内Wiki、Slack、議事録、設計書はLLMの学習データに含まれない
- **ハルシネーション** — 知らないことを「それっぽく」でっちあげてしまうことがある

つまり、LLM単体では**自社固有の質問に正確に答えられない**のです。

### 「検索してからLLMに渡す」というシンプルなアイデア

RAGの発想はとてもシンプルです。

1. ユーザーの質問に関連するドキュメントを**検索**する
2. 見つかったドキュメントをLLMのプロンプトに**注入**する
3. LLMがそのドキュメントを参考にして**回答を生成**する

これだけです。LLMを再学習させる必要はなく、ドキュメントを用意して検索できるようにするだけで、LLMが社内の専門家のように振る舞えるようになります。

## RAGの仕組み

### 検索 → コンテキスト注入 → 生成 のフロー

```
ユーザーの質問
  「新しい有給申請フローはどうなった？」
       │
       ▼
  ┌──────────┐
  │  検索     │  質問をベクトル化し、類似ドキュメントを検索
  │(Retrieval)│
  └────┬─────┘
       │  関連ドキュメント上位3件を取得
       ▼
  ┌──────────┐
  │コンテキスト│  検索結果をプロンプトに埋め込む
  │  注入     │  「以下の資料を参考に回答してください：...」
  └────┬─────┘
       │
       ▼
  ┌──────────┐
  │  生成     │  LLMが資料を踏まえて回答を生成
  │(Generate) │
  └──────────┘
       │
       ▼
  「2024年4月から有給申請フローが変更され、
   Slackの #hr-request チャンネルで...」
```

ポイントは、LLM自体は何も新しく学習していないということです。**検索で見つけた情報をプロンプトに貼り付けているだけ**。シンプルだからこそ、導入しやすく、メンテナンスもしやすいのです。

### Naive RAG vs Advanced RAG

RAGにも段階があります。

| | Naive RAG | Advanced RAG |
|---|-----------|-------------|
| **検索** | 単純なベクトル類似度検索 | ハイブリッド検索（ベクトル＋キーワード） |
| **前処理** | テキストを固定サイズで分割 | 文書構造を考慮した分割、メタデータ付与 |
| **後処理** | 検索結果をそのまま渡す | リランキング（検索結果の並び替え）、フィルタリング |
| **質問処理** | ユーザーの質問をそのまま使う | クエリ拡張・分解（質問を検索に適した形に変換） |
| **精度** | そこそこ | 高い |
| **実装難易度** | 低い | 中〜高 |

**まずは Naive RAG で始めて十分です。** 精度が足りないと感じたときに、Advanced RAG のテクニックを一つずつ追加していくのが実践的なアプローチです。

## 埋め込みモデル（Embedding）

### テキストをベクトルに変換する仕組み

RAGの「検索」を支えるのが**埋め込みモデル（Embedding Model）**です。テキストを数百〜数千次元のベクトル（数値の配列）に変換します。

```
「今日の天気は晴れです」
  → [0.023, -0.041, 0.187, ..., 0.092]  （1536次元の数値配列）

「本日は快晴でした」
  → [0.025, -0.038, 0.191, ..., 0.089]  （似た意味 → 似たベクトル）

「Pythonの例外処理について教えて」
  → [-0.112, 0.067, -0.034, ..., 0.241]  （違う意味 → 違うベクトル）
```

意味が近いテキストは**ベクトル空間上で近い位置**に配置されます。これにより、キーワードが完全一致しなくても、意味的に関連するドキュメントを見つけられるのです。

### 主要な埋め込みモデル

| モデル | 提供元 | 次元数 | 特徴 |
|--------|--------|--------|------|
| **text-embedding-3-small** | OpenAI | 1536 | コスト重視。日常用途に十分 |
| **text-embedding-3-large** | OpenAI | 3072 | 高精度。大規模検索向け |
| **BGE-M3** | BAAI | 1024 | オープンソース。多言語対応。ローカル実行可能 |
| **multilingual-e5-large** | Microsoft | 1024 | オープンソース。100言語以上対応。日本語に強い |

**選び方の目安：**
- 手軽に始めるなら **text-embedding-3-small**（API呼び出しだけで使える）
- コストゼロ・プライバシー重視なら **BGE-M3** や **multilingual-e5**（ローカル実行）
- 日本語ドキュメントが多いなら **multilingual-e5-large** がおすすめ

### チャンク戦略（テキストの分割方法）

ドキュメントをそのまま埋め込むと長すぎるため、適切なサイズに分割（チャンキング）する必要があります。

```
元のドキュメント（10,000文字）
  ↓ チャンキング
┌──────────┐ ┌──────────┐ ┌──────────┐
│ チャンク1  │ │ チャンク2  │ │ チャンク3  │ ...
│ 500文字   │ │ 500文字   │ │ 500文字   │
└──────────┘ └──────────┘ └──────────┘
   ↓              ↓              ↓
 ベクトル化      ベクトル化      ベクトル化
```

**チャンクサイズの考え方：**

| パラメータ | 小さい（200〜300文字） | 大きい（800〜1500文字） |
|-----------|----------------------|----------------------|
| **検索精度** | 高い（ピンポイントで見つかる） | やや低い（ノイズが混ざる） |
| **文脈の豊富さ** | 低い（前後の文脈が切れる） | 高い（まとまった情報が取れる） |
| **用途** | FAQ、定義の検索 | 技術文書、レポートの検索 |

**オーバーラップ（重なり）：** チャンク間で50〜100文字程度を重複させると、文の途中で切れて意味が失われるのを防げます。まずは**チャンクサイズ500文字、オーバーラップ50文字**で始めてみて、精度を見ながら調整するのが実践的です。

## ベクトルDBとは

### なぜ普通のDBでは駄目なのか

通常のデータベース（PostgreSQL, MySQL 等）は、完全一致やキーワード検索が得意です。しかしRAGでは**「意味が近いものを探す」**必要があります。

```
通常のDB検索:
  WHERE title = '有給申請'        ← 完全一致
  WHERE title LIKE '%有給%'       ← キーワード部分一致

ベクトル類似度検索:
  「休暇の取り方」で検索
  → 「有給申請フロー」がヒット    ← 意味的に近い
  → 「休暇届の書式」がヒット      ← キーワードは違うが関連
```

ベクトルDBは、数百万〜数億のベクトルの中から、**クエリに最も近いベクトルを高速に見つける**ことに特化したデータベースです。

### 主要なベクトルDB比較

#### Pinecone（フルマネージド・クラウド）

公式サイト: [pinecone.io](https://www.pinecone.io/)

- クラウドのフルマネージドサービス。インフラ管理不要
- スケーラビリティが高い。数十億ベクトルにも対応
- メタデータフィルタリング、名前空間によるデータ分離
- 無料枠あり（インデックス数制限）

#### Weaviate（オープンソース）

公式サイト: [weaviate.io](https://weaviate.io/)

- オープンソース（BSD-3ライセンス）。セルフホスト可能
- ベクトル検索＋キーワード検索のハイブリッド検索を標準搭載
- GraphQL APIでクエリ可能
- マネージドクラウド版（Weaviate Cloud）もあり

#### Chroma（軽量・ローカル・プロトタイプ向け）

公式サイト: [trychroma.com](https://www.trychroma.com/)

- **最も手軽に始められる**。`pip install chromadb` だけで使える
- インメモリ or ローカルファイルで動作。外部サーバー不要
- Python/JavaScript SDK が充実
- プロトタイプや小規模プロジェクトに最適

#### pgvector（PostgreSQL拡張）

公式サイト: [github.com/pgvector/pgvector](https://github.com/pgvector/pgvector)

- **既存の PostgreSQL にベクトル検索機能を追加**する拡張
- 新しいデータベースを導入する必要がない
- SQLでベクトル検索が書ける（`ORDER BY embedding <=> query_vector`）
- Supabase, Neon, Railway 等のマネージドPostgreSQLでも利用可能

### 比較テーブル

| 特徴 | Pinecone | Weaviate | Chroma | pgvector |
|------|----------|----------|--------|----------|
| **種別** | クラウド専用 | OSS + クラウド | OSS | PostgreSQL拡張 |
| **導入の手軽さ** | ○ | △ | ◎ | ○ |
| **スケーラビリティ** | ◎ | ○ | △ | ○ |
| **ハイブリッド検索** | ○ | ◎ | △ | ○ |
| **既存DBとの統合** | × | × | × | ◎ |
| **セルフホスト** | × | ○ | ○ | ○ |
| **ユースケース** | 大規模本番環境 | 本番〜中規模 | プロトタイプ | 既存PG環境 |
| **料金** | 従量課金 | 無料〜有料 | 無料 | 無料 |

**選び方の目安：**
- **まず試したい** → Chroma（インストールだけで動く）
- **すでにPostgreSQLを使っている** → pgvector（DB追加なしで導入）
- **本番環境でしっかり運用したい** → Pinecone or Weaviate
- **ハイブリッド検索が重要** → Weaviate

## 実践的なユースケース

### 社内ドキュメント検索

最も一般的なユースケースです。社内Wiki、Confluence、Notion のページをベクトル化し、自然言語で質問できるようにします。

```
社員: 「リモートワークの申請に必要な書類は？」
RAG:  社内規定から関連ドキュメントを検索 → LLMが要約して回答
```

### ナレッジベース構築

FAQ、マニュアル、過去の問い合わせ対応をベクトルDBに蓄積。新しい質問に対して過去の回答を参考に応答します。

### コードベースへの質問応答

リポジトリ内のソースコードやドキュメントをチャンク化して格納。「この関数の使い方は？」「エラーハンドリングのパターンは？」といった質問に、コードベースを参照して回答します。

### カスタマーサポートの自動化

製品ドキュメント、ヘルプセンター、過去のサポートチケットをRAGで検索し、顧客からの問い合わせに自動応答。人間のオペレーターへのエスカレーション判断もLLMに任せられます。

## 始め方（最小構成）

Chroma + OpenAI Embeddings で、最小限のRAGを実際に動かしてみましょう。

### 必要なもの

```bash
pip install chromadb openai
```

- Python 3.9以上
- OpenAI APIキー（埋め込みモデル用）

### 最小限のRAGコード

```python
import chromadb
from openai import OpenAI

# --- 1. セットアップ ---
openai_client = OpenAI()  # 環境変数 OPENAI_API_KEY を使用
chroma_client = chromadb.Client()  # インメモリで起動
collection = chroma_client.create_collection("my_docs")


# --- 2. ドキュメントの埋め込みと格納 ---
documents = [
    "有給休暇は入社6ヶ月後に10日付与されます。申請はSlackの#hr-requestチャンネルで行ってください。",
    "リモートワークは週3日まで可能です。事前に上長の承認が必要です。申請はFreeeから行います。",
    "交通費は月額3万円まで支給されます。定期代は6ヶ月分を一括申請してください。",
    "健康診断は年1回、会社負担で受診できます。提携クリニックの予約はTeamsカレンダーから。",
]


def get_embedding(text: str) -> list[float]:
    """OpenAI APIでテキストをベクトルに変換"""
    response = openai_client.embeddings.create(
        model="text-embedding-3-small",
        input=text,
    )
    return response.data[0].embedding


# ドキュメントをベクトル化して格納
for i, doc in enumerate(documents):
    collection.add(
        ids=[f"doc_{i}"],
        documents=[doc],
        embeddings=[get_embedding(doc)],
    )

print(f"{len(documents)}件のドキュメントを格納しました")


# --- 3. 検索 → コンテキスト注入 → 生成 ---
def ask(question: str, n_results: int = 2) -> str:
    """質問に対してRAGで回答を生成"""

    # (a) 質問をベクトル化して類似ドキュメントを検索
    results = collection.query(
        query_embeddings=[get_embedding(question)],
        n_results=n_results,
    )
    context = "\n".join(results["documents"][0])

    # (b) 検索結果をプロンプトに注入してLLMに渡す
    response = openai_client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": (
                    "あなたは社内情報に詳しいアシスタントです。"
                    "以下の参考資料のみに基づいて回答してください。"
                    "資料に載っていない場合は「わかりません」と答えてください。"
                ),
            },
            {
                "role": "user",
                "content": f"【参考資料】\n{context}\n\n【質問】\n{question}",
            },
        ],
    )
    return response.choices[0].message.content


# --- 4. 実行 ---
question = "在宅勤務は週何日までできますか？"
answer = ask(question)
print(f"\n質問: {question}")
print(f"回答: {answer}")
```

### 実行結果の例

```
4件のドキュメントを格納しました

質問: 在宅勤務は週何日までできますか？
回答: リモートワーク（在宅勤務）は週3日まで可能です。
     事前に上長の承認が必要で、申請はFreeeから行ってください。
```

「在宅勤務」というキーワードはドキュメントに含まれていませんが、「リモートワーク」のドキュメントが意味的に近いため正しくヒットしています。これがベクトル検索の力です。

### ここから先のステップ

この最小構成を起点に、以下のように拡張していけます。

| ステップ | やること | 目的 |
|---------|---------|------|
| 1 | ドキュメント数を増やす | 実際の社内情報を網羅 |
| 2 | チャンクサイズを調整する | 検索精度の向上 |
| 3 | Chroma を永続化モードに変更 | データの永続化（`chromadb.PersistentClient`） |
| 4 | pgvector や Weaviate に移行 | 本番運用への対応 |
| 5 | ハイブリッド検索を導入 | キーワード検索との併用で精度向上 |
| 6 | リランキングを追加 | 検索結果の精度をさらに向上 |
