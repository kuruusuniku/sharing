# 5. Claude Code を上手く使うためのベストプラクティス

![ベストプラクティス 4つの柱](assets/images/ch05-tips.svg)

最初は何を聞いても「すごい」と思っていました。でも使い込むうちに、AIの出力品質が自分の指示の出し方に大きく依存することに気づきました。同じモデルでも、聞き方ひとつで結果がまるで違う。この章は、その試行錯誤から得た知見をまとめたものです。

### たとえ話：道案内と同じ

プロンプティングは**道案内**と同じです。

「だいたい西に行って」と言われても、目的地にたどり着けない。でも「あの大きな木を左に曲がって、コンビニが見えたら右」と言われれば、迷わず着ける。

AIへの指示も同じ。「テストを書いて」ではなく「`src/auth/login.ts` の `validateToken` 関数について、期限切れトークンと不正なフォーマットのエッジケースのテストを vitest で書いて」。具体的なランドマーク（ファイル名、関数名、条件）を与えれば与えるほど、AI は正確な場所にたどり着けます。

Anthropic が公式に公開しているベストプラクティスガイドのポイントをまとめます。

> 原文: [Best Practices for Claude Code](https://www.anthropic.com/engineering/claude-code-best-practices)

## 最重要：コンテキストウィンドウを意識する

Claude Code の性能に最も影響するのは**コンテキストウィンドウ**（会話の蓄積量）です。

- 会話が長くなるほど、Claude の精度が下がる
- 関係ない話題が混ざると、重要な指示を見落とす
- **`/clear` でこまめにリセット**するのが基本

### /clear の発見 — 生産性が2倍になった瞬間

これは多くの人が見落としている、最もシンプルで効果的なテクニックです。

私は最初、1つのセッションで何時間も作業を続けていました。すると後半になるにつれて、Claude の回答が曖昧になり、さっき言った指示を忘れ、同じミスを繰り返すようになる。「モデルの性能が下がった？」と思っていました。

違いました。**コンテキストが汚染されていたのです。**

タスクが切り替わるたびに `/clear` を打つようにしただけで、体感の生産性が2倍になりました。新鮮なコンテキストで始めるとClaude の回答の精度が明らかに上がるのです。

あなたもこう思ったことありませんか？——「さっきまで優秀だったのに、急にポンコツになった」。それ、`/clear` 不足かもしれません。

## 5つの基本原則

### 1. Claude が自分で検証できるようにする

最もレバレッジの高いテクニックです。

```
❌ 「メールバリデーション関数を作って」
✅ 「validateEmail関数を作って。user@example.com → true、invalid → false、
    user@.com → false のテストケースも書いて、テストを実行して」
```

テスト・スクリーンショット・期待出力を渡すことで、Claude が自分でフィードバックループを回せるようになります。

### 2. まず調査、次に計画、最後にコード

```
調査（Plan Mode）→ 計画 → 実装（Normal Mode）→ コミット
```

- **Plan Mode**（`Shift+Tab` で切替）: ファイルを読むだけで変更しない安全モード
- まず理解してから計画を立て、計画に沿って実装する
- 小さな修正（タイポ修正、ログ追加）ならスキップしてOK

### 3. 具体的なコンテキストを与える

```
❌ 「テストを追加して」
✅ 「foo.py のログアウト時のエッジケースのテストを書いて。モックは使わないで」

❌ 「ログインのバグを直して」
✅ 「セッションタイムアウト後にログインが失敗する。src/auth/ のトークンリフレッシュを確認して。
    再現するテストを書いてから修正して」
```

- `@ファイル名` でファイルを直接参照
- 画像のコピペ・ドラッグ&ドロップも可能
- 既存パターンを指定：「HotDogWidget.php と同じパターンで作って」

### 4. CLAUDE.md を活用する（ただし短く）

```markdown
# CLAUDE.md の良い例
- ES modules を使う（CommonJS は使わない）
- テストは vitest で書く
- 型チェックは変更後に必ず実行する
```

**入れるべきもの：** Claude が推測できないコマンド、デフォルトと異なるルール、プロジェクト固有の制約

**入れないもの：** コードを読めばわかること、一般的な言語規約、頻繁に変わる情報

長すぎる CLAUDE.md は逆効果——指示が埋もれて無視されます。

### 5. サブエージェントで調査を委任する

```
「サブエージェントを使って、認証システムのトークンリフレッシュの仕組みを調べて」
```

サブエージェントは別のコンテキストで動くので、メインの会話を汚さずに大量のファイルを調査できます。

## セッション管理のコツ

| テクニック | 説明 |
|-----------|------|
| **`/clear`** | 無関係なタスク間でコンテキストをリセット |
| **`Esc`** | Claude を途中で止めて方向修正 |
| **`Esc × 2`** | 会話とコードを前の状態に巻き戻し |
| **`/compact`** | 会話を要約してコンテキストを節約 |
| **`claude --continue`** | 前回の会話を再開 |
| **`claude --resume`** | 過去の会話一覧から選んで再開 |

## よくある失敗パターン

| パターン | 問題 | 対策 |
|---------|------|------|
| **なんでもセッション** | 1つのセッションで無関係なタスクを混ぜる | タスクごとに `/clear` |
| **何度も訂正** | 同じ修正を何回も指示する | 2回失敗したら `/clear` して最初から良いプロンプトで |
| **CLAUDE.md 肥大化** | 指示が多すぎて無視される | 定期的に削る。Claude が既にできることは書かない |
| **検証なしで信頼** | もっともらしい実装だが実は動かない | テスト・検証手段を必ず用意する |
| **無限調査** | スコープを決めずに「調べて」と言う | 範囲を絞るか、サブエージェントに委任する |

## Hooks — 確実に実行させる自動化

CLAUDE.md の指示は「お願い」ですが、**Hooks は「強制」** です。特定のタイミングで必ずスクリプトを実行します。

```
# Claude にHookを作ってもらう例
「ファイル編集後に毎回 eslint を実行する Hook を書いて」
「migrations フォルダへの書き込みをブロックする Hook を書いて」
```

Hooks は `.claude/settings.json` に設定されます。`/hooks` コマンドで現在の設定を確認できます。

**使いどころ：**
- ファイル保存後にリンター/フォーマッターを自動実行
- 特定ディレクトリへの書き込みを禁止
- コミット前にテストを自動実行

## スケールする使い方

- **並列セッション**: 複数の Claude を同時に走らせる（Writer + Reviewer パターン）
- **非対話モード**: `claude -p "プロンプト"` でCI/スクリプトに組み込み
- **ファンアウト**: 大量ファイルへの一括処理をループで実行

## Q&A — よくある疑問

### Q: プロンプトが長すぎる場合は？ どこまで書けばいい？

**A:** 「長いプロンプト」自体は問題ありません。問題なのは「曖昧で長いプロンプト」です。5行でも具体的なら十分ですし、20行でもすべてが必要な情報なら問題ない。目安としては、「この情報がないと Claude が間違える可能性がある」ものだけ書く。逆に「なくてもClaude が推測できる」情報は省く。迷ったら、まず短いプロンプトで試して、結果が不十分なら情報を足す——この反復が最速です。

### Q: 英語で書いた方がいい？ 日本語だと精度が落ちる？

**A:** 実用上、日本語で問題ありません。Claude は多言語対応しており、日本語のプロンプトでも十分な精度が出ます。ただし、コード内のコメントや変数名は英語が自然ですし、エラーメッセージの検索は英語の方がヒットしやすい。私の使い分けは「指示は日本語、コード関連の固有名詞はそのまま英語」です。

### Q: 失敗したプロンプトの見分け方は？

**A:** 以下の兆候があれば、プロンプトを書き直すべきサインです：(1) Claude が「確認ですが...」と何度も聞き返してくる → 情報が足りない、(2) 全然違う方向の実装が返ってくる → ゴールが曖昧、(3) もっともらしいが動かないコードが返ってくる → 検証手段を与えていない。2回修正しても直らないなら、`/clear` して最初から書き直す方が速いです。
