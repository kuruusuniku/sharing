# 9. オープンLLMとローカル実行

クラウドAPIに月何万円も払い続けるのか？ ローカルで動かせるなら動かしたい。そう思ったのがオープンLLMへの入口でした。実際に試してみると、用途によってはクラウドモデルに引けを取らない性能が、コストゼロで手に入ります。

**オープンモデルは「自分の畑で野菜を育てる」ようなものです。** スーパーで買う（クラウドAPI）方が手軽だし品揃えも良い。でも自家栽培なら、何を育てるかは完全に自分で決められる。農薬（データ送信）の心配もない。収穫し放題でコストはタネ代（電気代）だけ。手間はかかるけど、一度環境を整えれば無限に使えます。

重みが公開されており、自分のマシンで動かせるLLMの世界です。クラウドに依存しない、プライバシー重視、コスト無制限という利点があります。

## 「Llamaの衝撃」——オープンLLMの転換点

2023年2月、MetaがLlama（初代）の重みをリークのような形で公開したとき、AI業界に激震が走りました。「あれ、GPT-3.5に近い性能のモデルが、自分のマシンで動くの？」と。それまでLLMは巨大テック企業だけが動かせるものだった。それが突然、誰でも触れるようになった。

続く2023年7月、MetaはLlama 2を正式にオープンウェイトで公開。これで流れが決まりました。世界中の研究者や企業が重みをベースに改良を加え、オープンLLMのエコシステムが爆発的に広がったのです。

**そしてDeepSeekの衝撃。** 2024年末〜2025年初頭にかけて、中国のDeepSeekが発表したモデルは業界を震撼させました。「中国のスタートアップが、フラッグシップモデルに匹敵する性能をオープンウェイトで、しかも圧倒的に少ない計算資源で実現した」——これはAI開発に莫大な資金が必要という常識を覆すものでした。MoEアーキテクチャの巧みな活用と、効率的な学習手法がその秘密です。

## MoE（Mixture of Experts）とは

最新のオープンLLMを語る上で欠かせないのが **MoE（Mixture of Experts）** アーキテクチャです。

### 従来型（Dense）との違い

```
Dense モデル（従来型）
  入力 → [全パラメータが処理] → 出力
  例: 27Bモデル → 27B全部が毎回計算

MoE モデル
  入力 → [ルーター] → [Expert 2, Expert 5 のみ活性化] → 出力
  例: 35Bモデル → 3Bだけが毎回計算、残りは休眠
```

MoEを直感的に理解するなら、**大きな病院**を想像してください。100人の専門医がいるけど、患者一人に対して診察するのは2〜3人の専門医だけ。でも病院としての総合力（知識量）は100人分ある。これがMoEの仕組みです。

**MoEの利点：**
- **知識容量は大きいまま、計算コストは小さい** — 総パラメータで知識を確保し、推論時は一部だけ使う
- **ローカル実行に有利** — 35Bモデルでも3Bしかアクティブにしないため、メモリと速度の両面で効率的
- **同サイズのDenseモデルを上回る性能** — より多くの知識を持ちながら軽量に動作

### GGUF と量子化

Hugging Face で「GGUF」というファイル形式をよく見かけます。これはローカル実行に最適化された形式です。

```
元のモデル（safetensors）
  → 高精度だがサイズが大きい（例: 27B → 約54GB）

量子化モデル（GGUF）
  → 精度を少し落としてサイズを圧縮
  → Q4_K_M: 約15GB（品質と圧縮のバランスが最良）
  → Q2_K: 約10GB（さらに圧縮、品質やや低下）
```

量子化は**写真のJPEG圧縮**に似ています。元の画像（FP16）は完璧だけど巨大。JPEG（量子化）にすると多少画質は落ちるけど、ファイルサイズが劇的に小さくなる。そして実用上は十分な品質が保たれる——特にQ4_K_Mあたりは、ほとんどの人が劣化に気づかないレベルです。

**LM Studio や Ollama は GGUF 形式をそのまま読み込めます。** Hugging Face で GGUF を探してダウンロードし、LM Studio にドラッグ＆ドロップするだけです。

### Spaces — AIデモを試す

[huggingface.co/spaces](https://huggingface.co/spaces) では、ブラウザ上でAIアプリを即座に試せます。

ダウンロードやインストール不要で、**モデルの性能を事前に確認できる**ので便利です。

### Transformers ライブラリ

Hugging Face が開発するオープンソースの Python ライブラリ。GitHub Star 16万以上。

## ローカル実行ツール

### LM Studio

**おすすめ度：最も手軽**

GUIでモデルをダウンロード・実行できるデスクトップアプリ。技術知識がなくても使えます。

- モデルの検索・ワンクリックダウンロード
- チャットUIで即座に試せる
- OpenAI互換のローカルAPIサーバーを立てられる（他ツールとの連携に便利）
- macOS / Windows / Linux 対応

```
LM Studio を起動
 → モデルを検索（例: "qwen3"）
 → ダウンロード
 → チャット or ローカルAPIサーバーを起動
```

初めてLM Studioでモデルが動いた瞬間は感動しますよ。「えっ、これが自分のマシンだけで動いてるの？」と。インターネット切っても動き続ける。何回聞いても課金されない。この体験は一度味わうとクセになります。

### Ollama

公式サイト: [ollama.com](https://ollama.com/)

CLIベースのローカルLLM実行ツール。コマンド一発でモデルをダウンロード・実行できます。

```bash
# インストール
curl -fsSL https://ollama.com/install.sh | sh

# モデルを実行（自動ダウンロード）
ollama run qwen3.6:27b

# OpenAI互換APIサーバーとしても動作
# http://localhost:11434/v1/chat/completions
```

- LM Studio より軽量・CLI向き
- Docker との相性が良い（サーバー用途）
- OpenAI 互換 API を自動で提供
- macOS / Linux / Windows 対応

**LM Studio vs Ollama：** GUIで試したいなら LM Studio、CLI/サーバー用途なら Ollama。

### vLLM

公式サイト: [docs.vllm.ai](https://docs.vllm.ai/)

高速推論に特化したLLMサービングエンジン。**本番環境でLLMをAPIとして提供する**ときの標準ツールです。

- PagedAttention による高効率なメモリ管理
- 複数リクエストの同時処理（バッチ推論）
- OpenAI 互換 API
- GPU サーバー向け

個人利用というより、チームやサービスとしてLLMを提供する場合に使います。

### OpenCode

CLIベースのAIコーディングツール。ローカルLLMをバックエンドに使えます。

- Claude Code のようなCLI体験をローカルLLMで実現
- LM Studio / Ollama のローカルAPIと組み合わせて使う
- オフラインでのコーディング支援が可能

## 必要なマシンスペック

ローカルLLMを快適に動かすには、**メモリ（RAM/VRAM）が最重要**です。

| モデルサイズ | 必要メモリ目安 | 体感 |
|-------------|--------------|------|
| 1B〜3B | 4GB〜 | 軽量。スマホでも動く。品質は限定的 |
| 7B〜8B | 8GB〜 | 実用最低ライン。簡単なタスクなら十分 |
| 14B〜27B | 16GB〜32GB | 日常利用に十分な品質 |
| 70B〜 | 48GB〜 | クラウドモデルに迫る性能。要ハイスペック |

### Mac mini がおすすめ

ローカルLLM用途で**Apple Silicon の Mac mini は最もコスパが良い選択肢**です。

**理由：**
- **ユニファイドメモリ** — CPUとGPUでメモリを共有。VRAM不足に悩まされない
- **Metal対応** — Apple GPU でLLM推論が高速
- **省電力** — 24時間つけっぱなしでも電気代が低い
- **コンパクト** — サーバーとして置いても場所を取らない

| 構成 | 価格目安 | 動かせるモデル |
|------|---------|--------------|
| Mac mini M4 16GB | 約10万円 | 7B〜14B が快適 |
| Mac mini M4 Pro 24GB | 約25万円 | 14B〜27B が快適 |
| Mac mini M4 Pro 48GB | 約35万円 | 27B〜70B も実用的 |

**おすすめ構成：** M4 Pro 24GB以上。Qwen3 27B が快適に動き、コーディング支援や設計レビューに十分使えます。

## ローカルLLMの使いどころ

| ユースケース | おすすめ |
|-------------|---------|
| 機密性の高いコードの分析 | ローカルLLM（データが外に出ない） |
| オフライン環境での作業 | ローカルLLM |
| 大量の繰り返し処理 | ローカルLLM（APIコストゼロ） |
| 最高品質の出力が必要 | クラウドモデル（Claude Opus / GPT） |
| 設計・レビューの壁打ち | ローカルLLM（気軽に何度でも） |

「何度聞いても無料」というのは、使い方を根本的に変えます。クラウドAPIだと「この質問、$0.05かかるけど聞く価値あるかな」と無意識にブレーキがかかる。ローカルなら遠慮なく壁打ちできる。この心理的な違いは思った以上に大きいです。

## Q&A

**Q: ローカルLLMは実用に耐える？**

A: 用途によります。正直に言うと、最高品質の出力が必要な場面ではクラウドモデル（Claude Opus等）にはまだ及びません。しかし、設計の壁打ち、コードレビュー、要約、翻訳といった日常タスクなら、Qwen 3.6-27B クラスのモデルで十分実用的です。特に「100回聞いてもコストゼロ」という点が、気軽に使い倒せるメリットになります。

**Q: MacとLinuxどっちがいい？**

A: **Mac（Apple Silicon）がおすすめ**です。理由はユニファイドメモリ。NVIDIAのGPUを使うLinuxマシンだと、VRAM容量がボトルネックになりがち（RTX 4090でも24GB）。Mac mini M4 Pro 48GBなら、その48GB全部をLLMに使えます。さらに省電力・静音で、24時間稼働のローカルAIサーバーとして最適。Linuxの方が良いのは、NVIDIAの高価なGPU（A100, H100）を使える場合、またはvLLMで高スループットのサービングをしたい場合です。

**Q: 量子化すると品質はどれくらい落ちる？**

A: Q4_K_M（4bit量子化）なら、体感ではほとんど劣化を感じません。ベンチマーク上は元のFP16モデルから2〜5%程度のスコア低下です。Q2_K（2bit）になると明確に劣化を感じる場面が出てきます。私の実感としては、Q4_K_Mで使って「あれ？ なんか変だな」と思ったことはほぼありません。まずはQ4_K_Mを試して、それでメモリが足りなければQ3を検討する、というステップがおすすめです。
