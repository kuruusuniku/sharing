# 8. クローズドLLMモデル概説

モデルが多すぎて選べない、という声をよく聞きます。私も最初はそうでした。でも実際に使い比べてみると、各社の哲学の違いが見えてきて面白い。この章では、主要なクローズドモデルの特徴と使い分けを整理します。

**モデル選びはレストラン選びに似ています。** 各社にはそれぞれの「シェフの哲学」があります。安全性を最優先に考える厨房、とにかくメニューの幅広さで勝負する店、Google検索という最強の食材倉庫を持つ店——どれが「正解」ということはなく、今日何を食べたいかで選ぶものです。

ソースコードや重みが非公開の、企業が提供するLLMサービスです。高性能だが、各社のAPI・アプリ経由でのみ利用できます。

## 各社の成り立ち——哲学を知ると選び方が変わる

モデルの特徴を理解するには、各社がどんな背景で生まれたかを知ると腑に落ちます。

**Anthropic** は、もともとOpenAIの安全性研究チームにいたメンバーが「AIの安全性をもっと真剣にやりたい」と独立して設立した会社です。だからこそClaudeは指示追従性が高く、「やってはいけないこと」の判断が比較的しっかりしている。開発者がコードを書くとき、余計なことをしない堅実さがあります。

**OpenAI** は「人類全体にAIの恩恵を」という理念でスタートし、当初はオープンソース志向でした。しかし GPT-3 以降、安全性とビジネスの両面から非公開路線に転換。良くも悪くも「AIブームの火付け役」であり、ChatGPTという巨大なユーザー基盤を持つことが最大の強みです。

**Google（DeepMind）** は、もともと世界最強のAI研究機関を抱えていた会社です。AlphaGo で世界を驚かせたチームがLLMに本腰を入れた。Google検索、YouTube、Gmailという膨大なデータとインフラが背景にあります。

**xAI** はイーロン・マスクが設立。X（旧Twitter）のリアルタイムデータをフル活用し、「検閲のないAI」を標榜しています。技術的には後発ながら、資金力とデータアクセスで急速に追い上げています。

## 主要モデル一覧

### Anthropic — Claude

公式サイト: [anthropic.com](https://www.anthropic.com/) / [claude.ai](https://claude.ai/)

| モデル | 特徴 |
|--------|------|
| **Opus 4.7** | 最高性能。複雑な推論・長文コード生成・設計に最適 |
| **Sonnet 4.6** | 速度と性能のバランス。日常の開発作業に最適 |
| **Haiku 4.5** | 高速・低コスト。軽量タスク・分類・要約向け |

**強み：** コーディング能力、長文コンテキスト（100K〜1Mトークン）、指示追従性の高さ、Claude Code によるエージェント的活用

**利用方法：** Claude Desktop（チャット）、Claude Code（CLI）、API

私がClaudeを主力にしている理由を一つ挙げるなら、「コードを書かせたときの安定感」です。他のモデルだと時々暴走して余計なことをしますが、Claudeは指示の範囲内で堅実に仕事をしてくれる。レストランで言えば、毎回同じクオリティの料理を出してくれる信頼できるシェフ、という感覚です。

### OpenAI — GPT

公式サイト: [openai.com](https://openai.com/) / [chatgpt.com](https://chatgpt.com/)

| モデル | 特徴 |
|--------|------|
| **GPT-5.5** | 最新フラッグシップ。推論力・マルチモーダル |
| **GPT-4o** | マルチモーダル（テキスト・画像・音声）の統合モデル |
| **o3 / o4-mini** | 推論特化（Reasoningモデル）。数学・論理・コード |

**強み：** マルチモーダル統合、ChatGPTの巨大なエコシステム、DALL-E による画像生成、プラグイン・GPTs

**利用方法：** ChatGPT（チャット）、Codex（CLI）、API

### Google — Gemini

公式サイト: [deepmind.google](https://deepmind.google/technologies/gemini/) / [gemini.google.com](https://gemini.google.com/)

| モデル | 特徴 |
|--------|------|
| **Gemini 3.1 Pro** | 最新フラッグシップ。複雑な多段階エージェントワークフローに最適化 |
| **Gemini 3.1 Flash** | 高速・低コスト。日常利用向け |
| **Gemini 3.1 Flash Live** | リアルタイム音声対話に対応 |

**強み：** 超長文コンテキスト、Google 検索との統合、Android/Google Workspace連携、Computer Use対応、ロボティクスモデル

**現状：** 仕事で本格的に使うにはまだ閾値を超えていない印象。日常的な調べものや要約には十分使える。今後の進化に期待。

### xAI — Grok

公式サイト: [x.ai](https://x.ai/) / [grok.com](https://grok.com/)

| モデル | 特徴 |
|--------|------|
| **Grok 4.3 Beta** | 200万トークンのコンテキスト（西側クローズドモデル最大）、16エージェント並列処理、ネイティブ動画理解 |
| **Grok 5**（予定） | 2026年Q2にベータ公開見込み |

**強み：** 超大規模コンテキスト、リアルタイム情報（X プラットフォーム統合）、マルチエージェント並列処理

**現状：** Grok 4.3 は SuperGrok Heavy（月額$300）限定のEarly Access。高額だが、コンテキスト長と並列処理能力は突出。エンジニアリング用途での実用性は今後に期待。

| 観点 | Claude | GPT | Gemini | Grok |

| **コーディング** | ◎ | ◎ | ○ | △ |
| **推論・分析** | ◎ | ◎ | ○ | ○ |
| **マルチモーダル** | ○ | ◎ | ◎ | ○ |
| **日本語** | ◎ | ◎ | ○ | △ |
| **エージェント活用** | ◎ | ○ | △ | △ |

この表を見ると「全部◎のモデルはないんだ」と思いませんか？ そうなんです。だからこそ、用途に応じて使い分けることに意味がある。レストランで言えば、寿司屋にパスタを頼まないのと同じことです。

## どう使い分けるか

- **仕事の開発・設計** → Claude（Claude Code / Opus）
- **日常の相談・調べもの** → ChatGPT or Gemini で十分
- **画像生成が必要** → ChatGPT（DALL-E）
- **リアルタイム情報** → Grok or Gemini
- **コスト重視の大量処理** → 各社の軽量モデル（Haiku, Flash, 4o-mini）

正直に言うと、私は8割の作業をClaudeでやっています。でもそれは私の作業がコーディング中心だからです。もしあなたの仕事がデザインや画像制作中心なら、GPTの方が合うかもしれない。「みんなが使っているから」ではなく、**自分の仕事の中心が何か**で選ぶのが一番です。

## Q&A

**Q: 結局どれを使えばいい？**

A: 迷ったら**まずClaudeかChatGPTの無料枠を1週間ずつ使って比べて**ください。普段の仕事で実際に聞きたいことを聞いてみるのが一番早い。レビュー記事やベンチマークスコアを読むよりも、自分の手で触った印象の方がずっと信頼できます。コーディング中心ならClaude、マルチメディア中心ならGPT、というのが現時点の私の結論です。

**Q: モデルが更新されたら移行は大変？**

A: 同じ会社内のモデル更新（例: Sonnet 4.5 → 4.6）なら、ほとんどの場合シームレスに移行できます。APIのエンドポイントやパラメータ構造は維持されるのが普通です。ただし、プロンプトの挙動が微妙に変わることはあるので、重要なプロンプトは更新後に動作確認してください。会社を跨いだ移行（Claude → GPT等）は、APIの形式が違うのでコード変更が必要ですが、OpenAI互換API（11章で解説）を使えば差し替えが楽になります。

**Q: 複数モデルの併用は現実的？**

A: 全然現実的です。むしろ私はそうしています。コーディングはClaude、画像生成が必要なときはChatGPT、ちょっとした調べものはGemini——という使い分けは普通にやっています。ただし、最初から複数契約すると月額が積み重なるので、まずはメインを一つ決めて、足りない部分だけ他のモデルで補う形がおすすめです。
