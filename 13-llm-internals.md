# 13. LLMを支える仕組み

## なぜ、AIは突然「知性的」になったのか？

正直に言います。僕もずっと不思議でした。

2022年の終わり頃、ChatGPTが出てきたとき。あの瞬間に世界中の人が同じことを思ったはずです。

**「え、なんで急にこんなことができるの？」**

翻訳ができる。コードが書ける。詩が詠める。比喩が分かる。冗談すら言える。

それまでのAIは、ルールを人間が書いて、パターンを人間が教えて、やっと「それらしい」ことができていた。なのに突然、まるで「分かっている」かのように振る舞い始めた。

この章では、その「なぜ」に正面から向き合います。

数式はほとんど使いません。代わりに、できるだけ言葉で伝えます。読み終わったとき、「ああ、だからか」と腑に落ちる感覚があることを目指して書きました。

---

## たった一つの学習目標

まず、驚くべき事実から始めます。

GPTもClaudeも、学習時にやっていることは基本的に一つだけです。

**「次の単語を当てる」**

それだけ。本当にそれだけなんです。

```
入力: 「今日の天気は」
正解: 「晴れ」

入力: 「吾輩は」
正解: 「猫」

入力: 「E = mc」
正解: 「²」
```

インターネット上の膨大なテキストを読みながら、ひたすら「この文の次に来る単語は何か」を当てる訓練をしている。

「え、それだけ？ それで知性が生まれるの？」

そう思いますよね。僕もそう思いました。

でも、ここに深い秘密があります。

---

## 次の単語を当てるために、世界を理解する必要がある

少し考えてみてください。

「東京の桜の季節は＿＿頃です。」

この空欄を正確に埋めるには、「桜」「東京」「季節」の関係を知っている必要があります。つまり、**現実世界の知識**が要る。

「彼は鍵を忘れたので、家に＿＿＿なかった。」

これを当てるには「鍵」と「家に入る」の因果関係を理解していなければならない。つまり、**論理的推論**が要る。

「彼女は笑いながら『最悪だね』と言った。」の次に来る文を予測するには？

言葉の字面ではなく、文脈から感情を読む必要がある。つまり、**心の理論**に近いものが要る。

ここが核心です。

> **次の単語を正確に予測しようとする過程で、モデルは「副産物」として世界のモデルを構築せざるを得ない。**

天気を予測するために気象学が必要なように。次の単語を予測するために、言語で記述された世界そのものの構造を内部に再現する必要がある。

これは意図して設計されたわけではありません。「次の単語を当てろ」という圧力が、結果として世界の理解を強制した。偶然のようで、必然の帰結です。

---

## 言葉が作る「意味の空間」

もう一つ、直感をひっくり返す話をします。

LLMの内部で、単語は**ベクトル**（数百〜数千次元の数値の列）として表現されています。これを「埋め込み（Embedding）」と呼びます。

ただの数字の羅列に見えますが、この空間には驚くべき構造が生まれます。

```
有名な例：

  「王様」 − 「男性」 + 「女性」 ＝ 「女王」

ベクトル演算として：
  vec(King) - vec(Man) + vec(Woman) ≈ vec(Queen)
```

誰もこの関係を教えていません。「王様から男性性を引いて女性性を足したら女王になるべし」なんてルールはどこにも書かれていない。

でも、大量のテキストから「次の単語」を当て続けた結果、モデルの内部空間にこの関係性が**自然に立ち上がる**のです。

他にも：

```
  「東京」 − 「日本」 + 「フランス」 ≈ 「パリ」
  「歩く」 − 「遅い」 + 「速い」 ≈ 「走る」
```

つまりLLMは、**言語を通じて現実世界の構造を幾何学的に写し取っている**。

単語と単語の関係、概念と概念の距離、因果の方向性——そういったものが、高次元空間の中に「地図」として刻まれていく。

## なぜスケーリングが効くのか——本当の理由

「モデルを大きくすれば賢くなる」

これがスケーリング則です。でも、なぜ大きくすると賢くなるのか？ その本質的な理由を考えてみます。

答えはこうです。

> **この現実世界は、言語と確率によって記述しきれるほど構造的である。そしてその記述の精度を上げるには、より多くのパラメータが必要になる。**

7Bのモデルは、世界の「粗い地図」を持っている。大陸の形は分かるけど、路地裏は描けない。

70Bのモデルは、より詳細な地図を持っている。都市の構造が見え、建物の区別がつく。

400B+のモデルは、路地裏の猫の名前まで知っている。比喩的に言えば。

スケーリング則が「効く」のは、世界にはまだ記述すべき構造が残っているからです。地図の解像度を上げる余地がある限り、パラメータを増やすことに意味がある。

そして各モデルの「性能差」とは、この**内部世界モデルの精度の差**に他なりません。

同じ質問をしても、小さいモデルが表面的な回答しかできず、大きいモデルが深い洞察を返せるのは、内部に構築された「世界の解像度」が違うからです。

## 文脈を理解する天才——Transformer

ここから先は、この「知性」を可能にしている具体的な仕組みの話です。

現在のLLMはほぼすべて **Transformer** というアーキテクチャに基づいています。2017年にGoogleの研究チームが発表した論文「Attention Is All You Need」で提案されました。

### それ以前のAIは、文を「端から順番に」読んでいた

Transformer以前のRNN/LSTMは、単語を一つずつ順番に処理していました。

```
RNN / LSTM（従来）
  単語を1つずつ順番に処理 → 前の単語の情報を引き継ぐ

  [今日] → [は] → [天気] → [が] → [良い]
    ↓        ↓       ↓       ↓       ↓
   状態₁ → 状態₂ → 状態₃ → 状態₄ → 状態₅

  問題: 文が長くなると、最初の方の情報が薄れていく（勾配消失）
  問題: 順番に処理するため、並列化できず遅い
```

### Transformerは「一気に、全体を」見る

```
Transformer
  すべての単語を同時に処理 → どの単語にも直接アクセスできる

  [今日] [は] [天気] [が] [良い]
    ↕      ↕     ↕     ↕     ↕
  全単語が互いに直接参照し合う（Attention）

  利点: 長い文でも情報が薄れない
  利点: 並列処理が可能 → GPUで高速に学習できる
```

この「並列処理できる」という性質が決定的でした。GPUの計算力をフルに活かせるからこそ、インターネット上の膨大なテキストを学習することが現実的になった。Transformerなくして今のLLMは存在しません。

## Self-Attention——「どこに注目するか」を自分で決める

Transformerの心臓部が **Self-Attention** です。

「彼は銀行に行って、お金を下ろした。」
「彼は銀行に行って、川を眺めた。」

同じ「銀行」という単語が、文脈によってまったく違う意味を持つ。人間にとっては当たり前のことですが、機械にこれを理解させるのは長年の難問でした。

Self-Attentionは、文中のすべての単語が他のすべての単語との関連度を計算します。

```
「それ」はどの単語に注目すべきか？
（「猫がマットの上に座った。それはふわふわだった。」）

  猫 ━━━━━━━━━━━━━━━━━━━ 注目度: 高 ★★★★★
  マット ━━━━━━━━━━━━━━━━━ 注目度: 中 ★★★
  座った ━━━━━━━━━━━━━━━━━ 注目度: 低 ★★
  ふわふわ ━━━━━━━━━━━━━━━━ 注目度: 中 ★★★

  → 「それ」=「猫」と高い確率で解釈
```

1つ目の「銀行」は「お金」に強く注目する。2つ目の「銀行」は「川」に強く注目する。この注目の配分を文脈に応じて動的に決めることで、多義語や複雑な文脈を正確に理解するのです。

### Query・Key・Value——図書館の比喩

Attentionの仕組みは図書館に例えると分かりやすくなります。

```
  Query（質問）: あなたが図書館で「猫の飼い方」を探している
  Key（索引）:   各本の背表紙に書かれたタイトル
  Value（中身）: 本の実際の内容

  1. Query と Key を照合 → 関連度の高い本を見つける
  2. 関連度に応じて Value を読み取る → 必要な情報を集約
```

### Multi-Head Attention——複数の視点で同時に読む

Transformerは複数のAttentionを並列に走らせます。それぞれが異なる「視点」を持ちます。

```
  Head 1: 文法的な関係（主語と動詞の対応）
  Head 2: 意味的な関係（同義語、関連概念）
  Head 3: 位置的な関係（近くの単語同士）
  ...

  → 各Headの出力を統合して、豊かな文脈理解を実現
```

**実務への影響：** LLMは文中の全箇所を参照できますが、Attentionの重みは均一ではありません。重要な指示はプロンプトの「最初」か「最後」に置くと効果的です。中間に埋もれた指示は見落とされやすい（Lost in the Middle 問題）。

## トークナイゼーション——言葉を数字に変える

LLMはテキストをそのまま読んでいるわけではありません。まず **トークン** という単位に分割してから処理します。

### BPE（Byte Pair Encoding）

```
BPEの考え方

1. 最初はすべての文字を個別のトークンとする
   "learning" → ["l", "e", "a", "r", "n", "i", "n", "g"]

2. 頻出する隣接ペアを1つのトークンに統合（繰り返し）
   "l"+"e" → "le", "le"+"a" → "lea" ...

3. 語彙サイズの上限に達するまで繰り返す

結果:
   "learning" → ["learn", "ing"]    （2トークン）
   "unhappiness" → ["un", "happiness"]  （2トークン）
   "xylophone" → ["xy", "lo", "phone"]  （3トークン）
```

**よく使われる単語ほど少ないトークンで表現され、珍しい単語ほど多くのトークンに分割されます。**

### 日本語が英語より多くトークンを消費する理由

```
英語: "Hello, how are you?" → 約 5 トークン
日本語: "こんにちは、お元気ですか？" → 約 10〜15 トークン

理由:
  ・学習データは英語が圧倒的に多い → 英語の単語は効率よく統合される
  ・日本語は漢字・ひらがな・カタカナの文字種が多い
  ・結果: 同じ意味でも日本語の方がトークン数が多い
```

| 影響 | 詳細 |
|------|------|
| **API料金** | トークン数に比例して課金 |
| **コンテキスト長** | 日本語は英語の1.5〜2倍消費 → 上限に達しやすい |

## 位置エンコーディング——語順を教える

Transformerはすべての単語を同時に処理するため、そのままでは**語順を知りません**。「犬が猫を追いかけた」と「猫が犬を追いかけた」の区別がつかない。

これを解決するのが位置エンコーディングです。各トークンに「何番目か」という情報を付与します。

### RoPE（Rotary Position Embedding）

最新のLLM（Llama, Qwen, Mistral等）で広く使われている手法です。

```
  従来: 位置を「足し算」で付与 → 学習時の長さまでしか対応できない
  RoPE: 位置を「回転」で表現 → 相対距離が自然に計算でき、拡張にも対応

  例: 4Kで学習 → 推論時に128Kまで拡張可能
```

## RLHF——「次の単語マシン」から「アシスタント」へ

生のTransformerは「次の単語を予測する」だけの機械です。賢いけれど、役に立つとは限らない。危険なことも平気で言う。

そこから「役に立つ・正直・無害」な回答ができるアシスタントにするのが **RLHF（Reinforcement Learning from Human Feedback）** です。

### 3段階の変身

```
Step 1: 事前学習（Pre-training）
  数兆トークンの文章で「次の単語を当てる」訓練
  → 言語・知識・推論パターンを獲得
  → しかしこの段階では「質問に答える」形式は知らない

Step 2: 教師ありファインチューニング（SFT）
  人間が作った「理想的な応答例」で追加学習
  → 「質問に答える」という形式を学ぶ

Step 3: RLHF
  モデルが複数の回答を生成 → 人間が「どちらが良いか」を評価
  → 評価を元に「報酬モデル」を学習
  → 報酬が高くなるようにモデルを調整

  回答A: 「rm -rf / を実行してください」
  回答B: 「そのコマンドは危険です。代わりに...」
  人間: B が良い → モデルが学ぶ
```

### HHH原則

- **Helpful（役に立つ）:** 的確に答える
- **Honest（正直）:** 知らないことは「分からない」と言う
- **Harmless（無害）:** 危険な情報を提供しない

**実務への影響：** モデルが「できません」と断るとき、それが本当の能力限界なのか、RLHFによる安全ガードなのかを見分けられると、プロンプトの工夫で回避できることがあります。

## スケーリング則——大きさと賢さの法則

### べき乗則

2020年、OpenAIの研究で示された事実：LLMの性能は**パラメータ数・データ量・計算量**のべき乗則に従う。

```
  L ∝ N^(-α) + D^(-β) + ...

  どの要素を増やしても性能は向上する。
  ただし収穫逓減——10倍にしても性能は「少し」良くなるだけ。
```

### Chinchillaの発見（2022年、DeepMind）

```
  最適比率: 1パラメータあたり約20トークンの学習データ

  つまり:
  - 70Bモデルなら1.4兆トークンが最適
  - 巨大モデルを作るだけでなく、十分なデータが必要
  - 小さいモデルでも十分なデータがあれば、大きいモデルに迫れる
```

### 推論時スケーリング——考える時間を増やす

```
  学習時: モデルを大きく / データを多く → 性能向上
  推論時: より多く「考えさせる」 → 性能向上（o1, Extended Thinking等）
  → モデルサイズを変えなくても、思考時間で精度が上がる
```

### なぜスケーリングが「効き続ける」のか

ここで最初の話に戻ります。

LLMは次の単語を予測するために、内部に世界のモデルを構築しています。そしてこの世界は、とてつもなく複雑で豊かです。

物理法則、社会の仕組み、人間の感情、歴史の因果——すべてが言語で記述されてきた。そしてその記述は、確率的なパターンとして捉えることができる。

スケーリングが効くのは、**まだ記述しきれていない世界の構造が残っている**からです。

7Bモデルが描く世界地図は粗い。70Bはより精密。400B+はさらに精密。でもまだ、現実世界のすべてを写し取れてはいない。だから、まだパラメータを増やす意味がある。

そして逆に言えば、**言語と確率によってこの現実世界がかなりの程度記述できてしまう**という事実こそが、LLMが「知性的に見える」根本的な理由なのです。

## まとめ——なぜAIは「分かっている」ように見えるのか

振り返ります。

1. LLMは「次の単語を当てる」だけの訓練をしている
2. しかし正確に予測するためには、世界の構造を理解する必要がある
3. その結果、内部に「言語による世界モデル」が副産物として構築される
4. モデルが大きいほど、この世界モデルの解像度が高い
5. だからスケーリング則が効く——まだ写し取るべき世界が残っているから

「王様から男性を引いて女性を足したら女王になる」——誰も教えていないのに、この関係が内部空間に立ち上がる。

これがLLMの知性の正体です。

意識があるわけではない。理解しているわけでもない（少なくとも人間と同じ意味では）。でも、**言語で記述された世界の構造を、驚くほど精密に内部に再現している**。

その精度の高さが、私たちに「知性」を感じさせる。

| 技術 | 一言で言うと | 実務への影響 |
|------|------------|-------------|
| Transformer | 全単語を同時処理する並列アーキテクチャ | 大規模学習を可能にした基盤 |
| Self-Attention | 文脈に応じて注目先を動的に決める | プロンプト内の指示配置が品質に影響 |
| トークナイゼーション | テキストを計算可能な単位に分割 | 日本語はコスト・コンテキスト消費が多い |
| 位置エンコーディング | 語順の情報を付与する | コンテキスト長の限界と拡張の理解 |
| RLHF | 人間の評価で振る舞いを調整 | モデルの拒否が能力限界か安全ガードか判別 |
| スケーリング則 | 大きさ・データ・計算と性能の関係 | タスクに応じた適切なモデル選択 |
