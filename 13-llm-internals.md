# 13. LLMを支える仕組み

ChatGPT や Claude を使うとき、裏側で何が起きているのか。ブラックボックスのままでも使えますが、仕組みを知ると**プロンプトの書き方・モデルの選び方・コストの見積もり**が格段に良くなります。

この章では、LLMを支える主要な技術を「なぜそうなっているのか」という視点で解説します。数式は最小限にとどめ、直感的な理解を目指します。

## なぜ仕組みを知るべきか

LLMの内部構造を知ることで、以下の判断が的確になります。

| 知識 | 活かせる場面 |
|------|-------------|
| Attention の仕組み | 長いプロンプトで重要な指示をどこに置くべきか |
| トークナイゼーション | コスト見積もり、日本語プロンプトのコスト感覚 |
| 位置エンコーディング | コンテキスト長の限界と、長文を扱う際の注意点 |
| RLHF | なぜモデルが「丁寧だが頑固」なことがあるのか |
| スケーリング則 | モデルサイズと性能のトレードオフの判断 |

仕組みを知る目的は研究者になることではなく、**道具としてのLLMをより上手に使うこと**です。

## Transformer アーキテクチャ

現在のLLMはほぼすべて **Transformer** というアーキテクチャに基づいています。2017年にGoogleの研究チームが発表した論文「Attention Is All You Need」で提案されました。

### 従来のRNN/LSTMとの違い

Transformer以前は、RNN（再帰型ニューラルネットワーク）やLSTMが自然言語処理の主流でした。

```
RNN / LSTM（従来）
  単語を1つずつ順番に処理 → 前の単語の情報を引き継ぐ

  [今日] → [は] → [天気] → [が] → [良い]
    ↓        ↓       ↓       ↓       ↓
   状態₁ → 状態₂ → 状態₃ → 状態₄ → 状態₅

  問題: 文が長くなると、最初の方の情報が薄れていく（勾配消失）
  問題: 順番に処理するため、並列化できず遅い

Transformer
  すべての単語を同時に処理 → どの単語にも直接アクセスできる

  [今日] [は] [天気] [が] [良い]
    ↕      ↕     ↕     ↕     ↕
  全単語が互いに直接参照し合う（Attention）

  利点: 長い文でも情報が薄れない
  利点: 並列処理が可能 → GPUで高速に学習できる
```

**これを知っていると：** Transformerが並列処理できるからこそ、大規模な学習が現実的になり、今のLLMが生まれたことが分かります。

### Attention メカニズム

Transformerの核心は **Attention（注意機構）** です。「文章のどこに注目するか」を学習する仕組みです。

例えば「猫がマットの上に座った。それはふわふわだった。」という文で、「それ」が何を指すかを理解するには、文中の他の単語との関係性を計算する必要があります。

```
「それ」はどの単語に注目すべきか？

  猫 ━━━━━━━━━━━━━━━━━━━ 注目度: 高 ★★★★★
  が ━━━━━━━━━━━━━━━━━━━ 注目度: 低 ★
  マット ━━━━━━━━━━━━━━━━━ 注目度: 中 ★★★
  の上に ━━━━━━━━━━━━━━━━━ 注目度: 低 ★
  座った ━━━━━━━━━━━━━━━━━ 注目度: 低 ★★
  それ ━━━━━━━━━━━━━━━━━━ （自分自身）
  は ━━━━━━━━━━━━━━━━━━━ 注目度: 低 ★
  ふわふわ ━━━━━━━━━━━━━━━━ 注目度: 中 ★★★

  → 「それ」=「猫」と高い確率で解釈
```

Attention は各単語について **Query（質問）・Key（索引）・Value（値）** の3つのベクトルを計算します。

```
Attention の流れ（直感的に）

  1. Query: 「それ」が「私は何を指している？」と質問する
  2. Key:   各単語が「私はこういう特徴を持っています」と名乗る
  3. 照合:  Query と Key の類似度を計算 → 注目度（重み）を決定
  4. Value: 注目度に応じて各単語の情報を集約 → 出力

  [Q: それ] × [K: 猫, マット, 座った, ...] → 重み → [V: 各単語の情報を加重平均]
```

### Self-Attention

LLMで使われるのは **Self-Attention** と呼ばれる仕組みです。「Self」とは、入力文の中で各単語が他の全単語との関係を計算することを意味します。

```
Self-Attention の処理イメージ

入力: [The] [cat] [sat] [on] [the] [mat]

各単語が他の全単語に対して注目度を計算:

        The   cat   sat   on   the   mat
  The  [0.1] [0.3] [0.1] [0.1] [0.2] [0.2]
  cat  [0.1] [0.2] [0.3] [0.1] [0.1] [0.2]
  sat  [0.1] [0.3] [0.1] [0.1] [0.1] [0.3]  ← sat は cat と mat に注目
  on   [0.1] [0.1] [0.2] [0.1] [0.1] [0.4]  ← on は mat に注目
  the  [0.2] [0.1] [0.1] [0.1] [0.1] [0.4]
  mat  [0.1] [0.1] [0.3] [0.2] [0.1] [0.2]

  → 各単語が文脈に応じた「意味」を持つ表現に変換される
```

さらに、Transformerは **Multi-Head Attention** を使います。複数の Attention を並列に走らせ、それぞれが異なる関係性（文法的な関係、意味的な関係など）を学習します。

```
Multi-Head Attention

  Head 1: 文法的な関係を学習（主語-動詞の対応など）
  Head 2: 意味的な関係を学習（同義語、関連語など）
  Head 3: 位置的な関係を学習（近くの単語同士の関係など）
  ...
  Head N: その他のパターン

  各Headの出力を結合 → 豊かな文脈表現を獲得
```

**これを知っていると：** プロンプトの書き方に影響します。LLMは文中の全箇所を参照できるため、重要な指示は「最初」か「最後」に置くと効果的です。中間に埋もれた指示は、Attentionの重みが分散して見落とされやすくなります（Lost in the Middle 問題）。

## トークナイゼーション

LLMはテキストをそのまま読んでいるわけではありません。まず **トークン** という単位に分割してから処理します。

### BPE（Byte Pair Encoding）

現在の主流は **BPE** というアルゴリズムです。

```
BPEの考え方

1. 最初はすべての文字を個別のトークンとする
   "learning" → ["l", "e", "a", "r", "n", "i", "n", "g"]

2. 頻出する隣接ペアを1つのトークンに統合
   "l"+"e" → "le"  （頻出ペア）
   "le"+"a" → "lea"  （頻出ペア）
   ...

3. これを語彙サイズの上限に達するまで繰り返す

結果:
   "learning" → ["learn", "ing"]    （2トークン）
   "unhappiness" → ["un", "happiness"]  （2トークン）
   "xylophone" → ["xy", "lo", "phone"]  （3トークン — 珍しい単語は分割が多い）
```

**よく使われる単語ほど少ないトークンで表現され、珍しい単語ほど多くのトークンに分割されます。**

### 日本語が英語より多くトークンを消費する理由

```
英語の場合:
  "Hello, how are you?" → 約 5 トークン

日本語の場合:
  "こんにちは、お元気ですか？" → 約 10〜15 トークン

なぜ？
  ・LLMの学習データは英語が圧倒的に多い
  ・BPEは頻出パターンを統合するため、英語の単語は効率よくまとまる
  ・日本語は漢字・ひらがな・カタカナの文字種が多く、
    英語ほど効率的な統合が起きにくい
  ・結果として、同じ意味の文章でも日本語の方がトークン数が多くなる
```

### コスト・コンテキスト長への影響

トークン数は**コストとコンテキスト長の両方に直結**します。

| 影響 | 詳細 |
|------|------|
| **API料金** | 入力・出力のトークン数に比例して課金される |
| **コンテキスト長** | モデルが一度に処理できるトークン数には上限がある（例: 200Kトークン） |
| **日本語のハンデ** | 同じ内容でも英語の1.5〜2倍のトークンを消費 → コストもコンテキスト消費も増加 |

**これを知っていると：** 日本語プロンプトのコスト見積もりが正確になります。また、コンテキスト長の上限に達しやすいことを意識して、プロンプトを簡潔にする工夫ができます。

## 位置エンコーディング

Transformerはすべての単語を同時に処理するため、**語順を知りません。** 「犬が猫を追いかけた」と「猫が犬を追いかけた」の区別がつかないのです。

これを解決するのが **位置エンコーディング（Positional Encoding）** です。

```
位置エンコーディングの考え方

  各トークンに「何番目か」という位置情報を付与する

  入力: [犬] [が] [猫] [を] [追いかけた]

  トークン表現:  [犬の意味] [が の意味] [猫の意味] ...
  位置情報:      [位置 1]   [位置 2]   [位置 3]   ...
  ─────────────────────────────────────────────
  最終入力:      [犬+位置1] [が+位置2] [猫+位置3] ...

  → これで語順の情報が保持される
```

### RoPE（Rotary Position Embedding）

最新のLLM（Llama, Qwen, Mistral など）で広く使われている手法が **RoPE** です。

```
RoPEの直感的な理解

  従来: 位置を「足し算」で付与
    → 学習時に見た長さまでしか対応できない

  RoPE: 位置を「回転」で表現
    → 2つのトークンの「相対的な距離」が自然に計算できる
    → 学習時より長いコンテキストにも（ある程度）対応可能

  例: コンテキスト長 4K で学習 → 推論時に 128K まで拡張
      YaRN, NTK-aware scaling などの手法で拡張精度を向上
```

**これを知っていると：** モデルの「コンテキスト長」の意味がより深く分かります。学習時のコンテキスト長と推論時の拡張限界は異なり、長すぎるコンテキストでは精度が低下する可能性があることを意識できます。

## RLHF（人間のフィードバックによる強化学習）

生のTransformerは「次の単語を予測する」だけの機械です。そこから「役に立つ・正直・無害」な回答ができるアシスタントにするのが **RLHF（Reinforcement Learning from Human Feedback）** です。

### 3段階の学習プロセス

```
ステップ1: 事前学習（Pre-training）
  ┌─────────────────────────────────────────────┐
  │ インターネット上の大量のテキストで学習          │
  │ → 「次の単語を予測する」能力を獲得             │
  │ → 言語の構造・知識・推論パターンを学ぶ          │
  │                                               │
  │ 学習データ: 数兆トークン（Web, 書籍, 論文...）  │
  │ 計算コスト: 数百万〜数千万ドル                  │
  └─────────────────────────────────────────────┘
              ↓
ステップ2: 教師ありファインチューニング（SFT）
  ┌─────────────────────────────────────────────┐
  │ 人間が作った「理想的な応答」の例で追加学習       │
  │                                               │
  │ 例:                                           │
  │   質問: 「Pythonでリストをソートするには？」     │
  │   理想: 「sorted()関数を使います。例えば...」   │
  │                                               │
  │ → 「質問に答える」形式を学ぶ                   │
  └─────────────────────────────────────────────┘
              ↓
ステップ3: RLHF
  ┌─────────────────────────────────────────────┐
  │ 1. モデルが複数の回答を生成                     │
  │ 2. 人間が「どちらが良い回答か」を評価           │
  │ 3. 評価データで「報酬モデル」を学習              │
  │ 4. 報酬モデルの評価が高くなるようにLLMを調整     │
  │                                               │
  │   回答A: 「rm -rf / を実行してください」         │
  │   回答B: 「そのコマンドは危険です。代わりに...」  │
  │   人間の評価: B の方が良い → 報酬モデルが学習   │
  └─────────────────────────────────────────────┘
```

### なぜ「役に立つ・正直・無害」な回答ができるのか

RLHFの評価基準は **HHH（Helpful, Honest, Harmless）** に基づいています。

- **Helpful（役に立つ）:** 質問に的確に答える。曖昧な返答を避ける
- **Honest（正直）:** 知らないことは「分からない」と言う。嘘をつかない
- **Harmless（無害）:** 危険な情報を提供しない。差別的な内容を避ける

RLHFによって、モデルはこれらの基準を「内面化」します。ただし完璧ではなく、時として過度に慎重になったり（「安全すぎる」回答）、自信満々に間違えたり（ハルシネーション）することがあります。

**これを知っていると：** モデルが「できません」と断るとき、それが本当の能力限界なのか、RLHFによる安全ガードなのかを見分けやすくなります。プロンプトの工夫で後者を回避できることがあります。

## スケーリング則

「モデルを大きくすれば賢くなる」——この経験則を理論的に裏付けたのが **スケーリング則（Scaling Laws）** です。

### モデルサイズ・データ量・計算量の関係

2020年にOpenAIが発表した研究で、LLMの性能は以下の3要素の**べき乗則（Power Law）** に従うことが示されました。

```
LLMの性能を決める3要素

  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
  │  パラメータ数  │     │  データ量     │     │  計算量      │
  │  (N)         │     │  (D)         │     │  (C)         │
  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
         │                   │                   │
         └───────────┬───────┘───────────────────┘
                     ↓
              モデルの性能（損失 L）

  L ∝ N^(-α) + D^(-β) + ...

  → どの要素を増やしても性能は向上する
  → ただし収穫逓減（10倍にしても性能は少しだけ向上）
```

### なぜ「大きいモデルほど賢い」のか

```
スケーリングの直感的な理解

  7Bモデル:
    パラメータが 70億個 → 知識の「引き出し」が 70億個
    → 基本的なタスクは可能だが、複雑な推論は苦手

  70Bモデル:
    パラメータが 700億個 → 引き出しが 10倍
    → より多くの知識を記憶し、複雑なパターンを学習できる

  400B+ モデル（Claude Opus, GPT-4 等）:
    さらに多くのパラメータ
    → 高度な推論、微妙なニュアンスの理解、複雑なコード生成
```

ただし、スケーリング則には重要な補足があります。

**Chinchillaの最適比率（2022年、DeepMind）：**

```
従来の考え方:
  「パラメータ数を増やせば性能が上がる」
  → 巨大なモデルを少ないデータで学習

Chinchillaの発見:
  「パラメータ数とデータ量は同じ比率で増やすべき」
  → 最適比率: 1パラメータあたり約20トークンの学習データ

  例: 70Bモデルなら 1.4兆トークンが最適
      → これより少ないデータで学習すると、モデルの潜在能力を引き出せない

影響:
  → 巨大モデルを作るだけでなく、十分なデータの確保が重要に
  → 小さいモデルでも十分なデータで学習すれば、
    大きいモデルに匹敵する性能を出せる場合がある
```

さらに最近では、**推論時のスケーリング（Test-time Compute Scaling）** も注目されています。

```
学習時のスケーリング:
  モデルを大きく / データを多く → 性能向上

推論時のスケーリング（o1, Claude の Extended Thinking 等）:
  推論時により多く「考えさせる」 → 性能向上
  → モデルサイズを変えなくても、推論に時間をかけることで
    より正確な回答が得られる
```

**これを知っていると：** モデル選択の判断が的確になります。「とにかく最大モデル」ではなく、タスクの複雑さに応じて適切なサイズを選べます。簡単なタスクに巨大モデルを使うのはコストの無駄であり、小さいモデルで十分なケースも多いです。

## まとめ

| 技術 | 一言で言うと | 実務への影響 |
|------|------------|-------------|
| Transformer | 全単語を同時処理する並列アーキテクチャ | GPUで大規模学習が可能に → 今のLLMが存在する理由 |
| Attention | 文中のどこに注目するかを学習する仕組み | プロンプト内の指示の配置が出力品質に影響する |
| トークナイゼーション | テキストを計算可能な単位に分割する | 日本語はコスト・コンテキスト消費が多い |
| 位置エンコーディング | 語順の情報を付与する仕組み | コンテキスト長の限界と拡張の理解 |
| RLHF | 人間の評価で「良い回答」を学習する | モデルの振る舞いの理由を理解できる |
| スケーリング則 | 大きさ・データ・計算の関係 | タスクに応じた適切なモデルサイズの選択 |

これらの仕組みを完全に理解する必要はありません。「こういう仕組みがあるから、こう動くのか」という直感を持っておくだけで、LLMをより効果的に活用できます。
