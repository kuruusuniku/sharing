# 10. 用途特化モデルの活用

ChatGPTに音声を文字起こしさせようとして失敗した経験、ありませんか？ 万能に見えるLLMにも得意不得意があります。音声認識、画像生成、ベクトル検索——これらは専用に訓練されたモデルの方が圧倒的に精度が高い。適材適所で組み合わせるのが実践のコツです。

**特化モデルは「電動工具」です。** スイスアーミーナイフ（汎用LLM）は便利だけど、壁に穴を開けたいなら電動ドリルの方が圧倒的に速いし正確。ノコギリで木を切りたいなら丸ノコを使う。一つのことを極めたツールには、万能ツールでは太刀打ちできない精度と速度があります。

汎用LLMだけがAIではありません。特定のタスクに特化したモデルを組み合わせることで、汎用モデルでは難しい精度や速度を実現できます。

## 音声 — Whisper

OpenAI が開発した音声認識モデル。**ローカルで無料で動かせる**のが最大の強みです。

### Whisperの誕生——68万時間の音声から学んだモデル

Whisperの開発ストーリーは興味深いです。OpenAIのチームは、インターネット上から**68万時間**もの多言語音声データ（字幕付き動画など）を集めて学習させました。68万時間というのは、約77年分の音声です。この「量で質を圧倒する」アプローチにより、従来の音声認識モデルが苦手だった多言語対応・ノイズ耐性・アクセント対応を一気に克服しました。しかもモデルをオープンソースで公開したため、誰でもローカルで無料で使えるようになった。この判断がなければ、私たちが手軽に高精度な文字起こしを使える世界は来ていなかったかもしれません。

### 特徴

- 多言語対応（日本語もOK）
- 高精度な文字起こし
- ローカル実行可能（APIコストなし）
- タイムスタンプ付きの出力が可能

### モデルサイズ

| モデル | パラメータ | 精度 | 速度 |
|--------|----------|------|------|
| tiny | 39M | △ | 最速 |
| base | 74M | ○ | 速い |
| small | 244M | ○ | 普通 |
| **medium** | 769M | **◎** | やや遅い |
| large-v3 | 1.5B | ◎ | 遅い |

**おすすめ：medium**。精度と速度のバランスが最も良い。`word_timestamps` オプションを有効にすると、単語単位のタイムスタンプが取れます。

### 使い方

```bash
# インストール
pip install openai-whisper

# 文字起こし
whisper audio.mp3 --model medium --language ja

# 単語単位のタイムスタンプ付き
whisper audio.mp3 --model medium --language ja --word_timestamps True
```

### 活用例

- 会議の議事録作成
- 講義・セミナーの文字起こし
- ポッドキャスト・動画の字幕生成
- 音声メモのテキスト化

私自身、講義音声の文字起こしにWhisperをかなり使っています。1時間の音声でも、M4 Macなら数分で処理が終わる。これをクラウドAPIでやったら結構な金額になるので、ローカル実行のありがたみを実感する瞬間です。

## 画像生成

| モデル/サービス | 種類 | 特徴 |
|---------------|------|------|
| **DALL-E 3** | クラウド | ChatGPT から呼べる。手軽 |
| **Midjourney** | クラウド | 高品質なアート生成。スタイル統一しやすい |
| **Stable Diffusion** | ローカル可 | オープンソース。カスタマイズ自由 |
| **FLUX** | ローカル可 | 新世代。高品質かつ高速 |

手軽に始めるなら **ChatGPT（DALL-E）** 、品質を求めるなら **Midjourney**、自由にカスタマイズしたいなら **Stable Diffusion / FLUX** をローカルで。

## 埋め込みモデル（Embedding）

テキストをベクトル（数値の列）に変換するモデル。**検索・類似度計算・RAG の基盤**になります。

| モデル | 特徴 |
|--------|------|
| **text-embedding-3-small/large** | OpenAI のAPI。高性能で手軽 |
| **BGE-M3** | オープンソース。多言語対応 |
| **multilingual-e5** | 多言語の埋め込みに強い |

### 何に使うのか

```
テキスト → 埋め込みモデル → ベクトル → 類似度検索
```

埋め込みモデルは**翻訳家**のような存在です。人間の言葉をコンピュータが「近い・遠い」を判断できる数値の世界に翻訳してくれる。「有給休暇」と「年次休暇」はキーワードとしては別物だけど、意味的には近い——その「意味の近さ」を数値で測れるようにするのが埋め込みモデルの仕事です。

- **社内ドキュメント検索** — 自然文で検索できる
- **RAG（検索拡張生成）** — LLM に関連文書を渡して回答精度を上げる
- **重複検出** — 似たチケットや文書を自動検出

## コード特化モデル

| モデル | 特徴 |
|--------|------|
| **Qwen Coder** | Qwen ベースのコーディング特化。ローカルで動く |
| **DeepSeek Coder** | コード生成・補完に特化 |
| **CodeGemma** | Google のコード特化軽量モデル |
| **StarCoder 2** | BigCode プロジェクトのオープンモデル |

汎用モデルでもコードは書けますが、コード特化モデルは**補完速度が速く、トークン効率が良い**ため、エディタの補完バックエンドとして有用です。

## 使い分けの考え方

汎用LLMが万能に見えますが、特化モデルを組み合わせた方が良い場面は多い：

- **音声 → テキスト** → Whisper（汎用LLMでは不可能）
- **テキスト → 画像** → DALL-E / SD（汎用LLMでは不可能）
- **大量の文字起こし** → Whisper ローカル（APIコストゼロ）
- **リアルタイムのコード補完** → コード特化モデル（低レイテンシ）
- **検索基盤** → 埋め込みモデル（汎用LLMでは非効率）

**汎用LLM は「考える」担当、特化モデルは「変換する」担当** と捉えると、使い分けがクリアになります。

これは会社組織に似ています。社長（汎用LLM）が判断・指示を出し、経理部（埋め込みモデル）、翻訳部（Whisper）、デザイン部（画像生成）がそれぞれの専門作業をこなす。全部社長一人でやろうとしたら破綻しますよね。

## Q&A

**Q: 汎用モデルだけで全部やるのはダメ？**

A: ダメではないですが、「できること」と「上手くできること」は違います。例えばClaude OpusにMP3ファイルを渡して文字起こしを頼むこと自体は可能ですが、Whisperの方が精度が高く、ローカルで動くのでコストもゼロです。汎用モデルだけで済ませられるなら複雑さは減りますが、精度・速度・コストのどれかで妥協することになります。「まず汎用モデルで試して、品質が足りなければ特化モデルを導入する」というステップが現実的です。

**Q: 特化モデルの組み合わせ方のコツは？**

A: パイプライン（直列処理）で考えると整理しやすいです。例えば「会議の議事録から要約とアクションアイテムを抽出する」なら：(1) Whisperで音声→テキスト、(2) Claude/GPTでテキスト→要約＋アクションアイテム。各ステップの入出力を明確にして、それぞれに最適なモデルを当てはめる。一つのモデルに全部やらせようとすると、どこかで精度が落ちます。パイプラインの各段で「ここは何が得意か」を考えるのがコツです。
