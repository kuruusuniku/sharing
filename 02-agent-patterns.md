# 2. Anthropic 公式 AI エージェントパターン

![5つのワークフローパターン](assets/images/ch02-patterns.svg)

AIに1回指示を出して完璧な結果が返ってくることなんて、正直ほぼありません。何回もやり取りして、修正して、また確認して。じゃあそのやり取り自体を自動化できないか？ そう考えたとき、Anthropicが体系化したパターンがそのまま答えになりました。

### なぜ Anthropic はこのガイドを公開したのか

Anthropic のエンジニアたちは、外部の開発者が Claude API を使ってエージェントを構築する様子を見ていました。そこで気づいたことがあります——**みんな、複雑にしすぎている**。

LangChain の長大なチェーン、独自フレームワークの層を何重にも重ね、過剰に抽象化されたアーキテクチャ。結果として、デバッグが困難で、コストが高く、実はシンプルなプロンプト1発で解決できる問題に複雑なパイプラインを構築してしまう。

「もっとシンプルに作れるはずなのに」——そう感じた Anthropic は、自社のカスタマーサクセスチームが実際に顧客と一緒に構築してきた経験を体系化して公開しました。それがこの「Building Effective Agents」です。

### たとえ話：パターンは料理のレシピ

5つのパターンを聞くと「全部理解してから使わないと」と思ってしまうかもしれません。でも、**料理のレシピ**だと思ってください。

カレーを作るのに、フレンチのソース技法を全部マスターする必要はない。まず一つのレシピを覚えて、作ってみて、「もうちょっとこうしたい」と思ったときに次のテクニックを学べばいい。

パターンも同じです。まずはエバリュエーター（レビューさせるだけ）から始めて、慣れたら組み合わせる。いきなり全部盛りにしない。**シンプルなものが動いてから、複雑さを足す。**

Anthropic は「[Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)」という公式ガイドで、AI エージェントの設計パターンを体系化しています。

重要な原則：**「複雑さは実証可能な改善があるときのみ追加する」**

## ワークフローパターン一覧

### 1. プロンプトチェーニング（Prompt Chaining）

タスクを順番に処理するパイプライン。各ステップの出力が次の入力になります。

```
[タスク] → LLM① → ゲート判定 → LLM② → [結果]
```

**例:** マーケティングコピー作成 → 別言語に翻訳

### 2. ルーティング（Routing）

入力を分類して、専門化された処理に振り分けます。

```
[入力] → 分類LLM → ┬→ 一般質問 → LLM-A
                     ├→ 返金対応 → LLM-B
                     └→ 技術サポート → LLM-C
```

**例:** カスタマーサポートの問い合わせ分類

### 3. 並列化（Parallelization）

同じタスクを複数の視点で同時処理し、結果を集約します。

```
         ┌→ LLM（視点A）→┐
[タスク] ┼→ LLM（視点B）→┼→ [集約]
         └→ LLM（視点C）→┘
```

**例:** コードの脆弱性を複数のプロンプトで同時検査

### 4. オーケストレーター・ワーカー（Orchestrator-Workers）

中央のLLMがタスクを**動的に**分解し、ワーカーに委任して結果を統合します。

```
[タスク] → オーケストレーター → ┬→ ワーカー①
                                 ├→ ワーカー②  → [統合]
                                 └→ ワーカー③
```

**ポイント:** 並列化との違いは、サブタスクが事前に決まっておらず、入力に応じて動的に決定される点です。

**例:** 複数ファイルにまたがる複雑なコード変更

### 5. エバリュエーター・オプティマイザー（Evaluator-Optimizer）

**これが「クリエイター＆レビューワー」に相当するパターンです。**

1つのLLMが成果物を**生成**し、別のLLMが**評価・フィードバック**を行い、基準を満たすまでループします。

```
[タスク] → 生成LLM → 成果物 → 評価LLM → 合格？ ─Yes→ [完了]
              ↑                            │
              └──── フィードバック ←──No──┘
```

**なぜ別のLLMで評価するのか？**
研究によると、LLMは自分の出力を過信する傾向があります。独立した評価者を懐疑的な視点で動かすことで、生成者が見落とす問題を検出できます。

**例:** 文学翻訳（ニュアンスの改善ループ）、複合的な検索タスク

## エージェント（自律型）

上記のワークフローとは別に、**エージェント**はLLMがツールを使いながら自律的にタスクを遂行するパターンです。

```
[指示] → LLM → ツール実行 → 結果観察 → LLM → ツール実行 → ... → [完了]
```

各ステップで環境からのフィードバック（ツール結果、コード実行結果）を得ることで、正確性を保ちます。Claude Code 自体がこのパターンで動作しています。

## まずはエバリュエーターから始めよう

5つのパターンを紹介しましたが、**最初に取り入れるべきは間違いなくエバリュエーター・オプティマイザーです。**

理由はシンプルです：

- **導入コストが最も低い** — 既存のワークフローに「レビューさせる」ステップを足すだけ
- **効果が最も実感しやすい** — AIの出力品質が目に見えて上がる
- **失敗のリスクが低い** — 最悪でも「レビューで何も見つからなかった」で済む

具体的には、Claude に何かを作らせたあと、**別のセッションで「これをレビューして」と頼む**だけです。同じAIでも、生成モードとレビューモードでは視点が変わります。

```
# セッション1: 生成
> このAPIのバリデーションロジックを実装して

# セッション2: レビュー（別セッションで実行）
> src/validation.ts をレビューして。
> エッジケース、セキュリティ上の問題、テストの漏れを指摘して。
```

これに慣れたら、次のステップとして：
- チェーニング（処理を段階的に分ける）
- 並列化（複数視点で同時チェック）
- オーケストレーター（タスクを動的に分解する）

と段階的に進めていくのがおすすめです。**いきなり複雑なパターンに手を出す必要はありません。**

## 実務での活用ポイント

1. **シンプルに始める** — まずはエバリュエーター。単一のLLM呼び出しで済むなら、それ以上は不要
2. **段階的に複雑さを追加** — 効果が測定できたときだけパターンを導入する
3. **ツールの品質が重要** — エージェントの性能はツールの設計品質に大きく依存する

あなたもこう思ったことありませんか？——「パターンは分かったけど、実際どれから手を付ければ？」。答えは明確です。**まず1つ、エバリュエーターを今日のタスクで試す。** それだけです。

## Q&A — よくある疑問

### Q: どのパターンから始めるべき？

**A:** エバリュエーター・オプティマイザー一択です。理由は、既存のワークフローを壊さずに「レビューステップを追加するだけ」で効果が出るから。今やっている作業にそのまま足せます。他のパターンは「必要だと感じてから」で十分です。

### Q: 複雑にしすぎないコツは？

**A:** 「今のシンプルなやり方で、具体的に何が困っているか？」を言語化してください。困っていない問題を解決するためにパターンを導入すると、確実に過剰設計になります。Anthropicのガイドが繰り返し言っているのはまさにこれ——「複雑さは実証可能な改善があるときのみ追加する」。

### Q: エバリュエーターは本当に効く？ 同じAIに聞いても同じ回答にならない？

**A:** なります——**同じセッションで**聞くと。だから別セッションで聞くのがポイントです。同じ LLM でも、「生成」モードと「レビュー」モードではプロンプトが異なるため、視点が変わります。実際にやってみると、生成時には気づかなかったエッジケースやセキュリティ問題が指摘されて驚くはずです。

### Q: フレームワーク（LangChain等）は使うべき？

**A:** Anthropic の公式見解は「必要になるまで使うな」です。フレームワークは抽象化レイヤーを追加するため、デバッグが困難になり、動作を理解しにくくなります。まずは素のAPI呼び出しやClaude Code で十分。フレームワークが必要になるのは、複数のパターンを組み合わせて本番運用する段階になってからです。
