# 2. Anthropic 公式 AI エージェントパターン

Anthropic は「[Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)」という公式ガイドで、AI エージェントの設計パターンを体系化しています。

重要な原則：**「複雑さは実証可能な改善があるときのみ追加する」**

## ワークフローパターン一覧

### 1. プロンプトチェーニング（Prompt Chaining）

タスクを順番に処理するパイプライン。各ステップの出力が次の入力になります。

```
[タスク] → LLM① → ゲート判定 → LLM② → [結果]
```

**例:** マーケティングコピー作成 → 別言語に翻訳

### 2. ルーティング（Routing）

入力を分類して、専門化された処理に振り分けます。

```
[入力] → 分類LLM → ┬→ 一般質問 → LLM-A
                     ├→ 返金対応 → LLM-B
                     └→ 技術サポート → LLM-C
```

**例:** カスタマーサポートの問い合わせ分類

### 3. 並列化（Parallelization）

同じタスクを複数の視点で同時処理し、結果を集約します。

```
         ┌→ LLM（視点A）→┐
[タスク] ┼→ LLM（視点B）→┼→ [集約]
         └→ LLM（視点C）→┘
```

**例:** コードの脆弱性を複数のプロンプトで同時検査

### 4. オーケストレーター・ワーカー（Orchestrator-Workers）

中央のLLMがタスクを**動的に**分解し、ワーカーに委任して結果を統合します。

```
[タスク] → オーケストレーター → ┬→ ワーカー①
                                 ├→ ワーカー②  → [統合]
                                 └→ ワーカー③
```

**ポイント:** 並列化との違いは、サブタスクが事前に決まっておらず、入力に応じて動的に決定される点です。

**例:** 複数ファイルにまたがる複雑なコード変更

### 5. エバリュエーター・オプティマイザー（Evaluator-Optimizer）

**これが「クリエイター＆レビューワー」に相当するパターンです。**

1つのLLMが成果物を**生成**し、別のLLMが**評価・フィードバック**を行い、基準を満たすまでループします。

```
[タスク] → 生成LLM → 成果物 → 評価LLM → 合格？ ─Yes→ [完了]
              ↑                            │
              └──── フィードバック ←──No──┘
```

**なぜ別のLLMで評価するのか？**
研究によると、LLMは自分の出力を過信する傾向があります。独立した評価者を懐疑的な視点で動かすことで、生成者が見落とす問題を検出できます。

**例:** 文学翻訳（ニュアンスの改善ループ）、複合的な検索タスク

## エージェント（自律型）

上記のワークフローとは別に、**エージェント**はLLMがツールを使いながら自律的にタスクを遂行するパターンです。

```
[指示] → LLM → ツール実行 → 結果観察 → LLM → ツール実行 → ... → [完了]
```

各ステップで環境からのフィードバック（ツール結果、コード実行結果）を得ることで、正確性を保ちます。Claude Code 自体がこのパターンで動作しています。

## 実務での活用ポイント

1. **シンプルに始める** — 単一のLLM呼び出しで済むなら、ワークフローは不要
2. **段階的に複雑さを追加** — 効果が測定できたときだけパターンを導入する
3. **ツールの品質が重要** — エージェントの性能はツールの設計品質に大きく依存する
